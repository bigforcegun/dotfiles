# MCP Stack — ресёрч

Разбор кандидатов на «централизованный менеджер MCP под мультиагентную систему» и обоснование финального выбора (**mcp-proxy + rulesync**). Состояние экосистемы — середина 2026.

---

## Постановка задачи

Нужен способ из одного места управлять всеми MCP в системе, поддерживая разные агентские системы (Claude Code, Cursor, Codex, OpenCode, Kilo), с прицелом на:
per-project включение/исключение, отсутствие утечки процессов, tool-level фильтрацию, конфиг в git, native/видимые конфиги, без резидентного Node.

---

## Две принципиально разные категории

Весь рынок делится на два архетипа, и их **нельзя совмещать без размена**:

1. **Config-менеджеры / синкеры** — пишут конфиги в каждый клиент из одного источника. Рантайма в пути запроса нет. Но каждый клиент сам запускает свои серверы → утечка возвращается, фильтрации нет.
2. **Gateway / aggregator** — единый эндпоинт, к которому ходят клиенты; владеет процессами, фильтрует, но встаёт прослойкой в путь.

**Ключевая правда:** «чистый менеджмент конечных конфигов» и «общий процесс + tool-фильтр + live-пресеты» — взаимоисключающи, потому что последнее требует рантайм-эндпоинта.

## Кандидаты

### mcpm.sh (pathintegral-institute) — ❌ отвергнут
- **Что:** CLI-менеджер MCP с глобальным реестром, профилями (виртуальные теги на серверах), Router, авто-детект клиентов. Python. Конфиг — плоские JSON (`servers.json`, `profiles_metadata.json`) в config-dir.
- **Плюс:** профили, мультиклиент, файловый конфиг.
- **Почему отвергнут (главное):** при применении сервера/профиля к клиенту mcpm **не пишет native-запись**, а вписывает враппер `{command: "mcpm", args: ["run", X]}` / `["profile","run",X]`. При старте клиент спавнит **Python-процесс mcpm** (FastMCP-прокси), который встаёт **второй прослойкой** перед реальным сервером. То есть это скрытый роутер, а не менеджер конфигов. Нарушает C2 (native/видимое) и C3 (Node/Python-резидент per-клиент).
- **Доп:** оперирует **по клиенту целиком**, не по проекту — F2 не закрывает.

### mcpup (mohammedsamin) — ⚠️ частично, не хватает per-project
- **Что:** чистый config-синкер. Один канонический `~/.mcpup/config.json` → native-записи в 13 клиентов (Claude Code, Cursor, Codex, OpenCode, …). Ownership-aware (не затирает чужое). Поддерживает remote HTTP/SSE.
- **Плюс:** ровно «менеджит конечные конфиги», без враппера; CLI, не демон; все нужные агенты.
- **Почему не финал:** **global-only**, проектного скоупа нет (F2 ✗). Для глобального базлайна годится, для per-project — нет.

### mcp-config-manager (holstein13) — ⚠️ как mcpup
- Интерактивный CLI, пресеты, enable/disable, синк Claude/Gemini. Тоже глобально-ориентирован, per-project не решает.

### MetaMCP (metatool-ai) — ❌ отвергнут
- **Что:** самый мощный gateway/aggregator. Docker + web-UI, иерархия Servers → Namespaces → Endpoints, middleware («filter inactive tools»), override имён/описаний тулзов, idle-session pooling (лучший lifecycle).
- **Плюс:** богатейшая tool-level фильтрация и per-namespace эндпоинты (close to per-project).
- **Почему отвергнут:** конфиг живёт в **Postgres + UI**, файлового источника нет → ломает C1 (IaC). Тяжёлый (Postgres). Это «БД с UI», а не текст в git.

### MCPJungle (mcpjungle) — ❌ отвергнут
- **Что:** self-hosted gateway, register-once → один эндпоинт, Tool Groups (подмножество тулзов на отдельный эндпоинт), access control.
- **Плюс:** tool-level через Tool Groups; группы описываются JSON-файлом.
- **Почему отвергнут:** авторитетное хранилище — **БД** (SQLite/Postgres). JSON только на вход (`register -c`), нет идемпотентного apply/diff → дрейф. Полу-IaC.

### Docker MCP Gateway — ❌ отвергнут
- Оркестрация серверов в Docker, креды, access control. Файлово-ish, но Docker-тяжёлый; против C3/C4.

### TBXark/mcp-proxy — ✅ выбран как рантайм-слой
- **Что:** один Go-бинарь, один `config.json`, без БД и UI. Агрегирует серверы, отдаёт каждый по своему URL `/<name>/{sse|mcp}`. Per-server `toolFilter` (allow/block). `--config <path|url>`, `--expand-env`.
- **Процессы (по исходникам):** на старте создаёт **один shared child** на сервер; много HTTP-клиентов шарят его; на SIGTERM gracefully `Close()`. Дети в дереве демона → `pkill` убивает всё. Решает F6.
- **Плюс:** C1 (один файл в git), C2 (native), C3 (Go, не Node), C4 (unix-way), F5 (`toolFilter`), F6 (процессы).
- **Минус:** нет авто-рестарта упавшего child (отдаём launchd `KeepAlive`); **не мёржит** серверы (каждый — свой URL, нет единого merged-эндпоинта); статический фильтр.
- **Роль в финале:** рантайм всего общего флота.

### 1mcp (1mcp-app/agent) — ❌ отвергнут, но был финалистом
- **Что:** Node-аггрегатор. Один инстанс, один эндпоинт, subset по query: `?tags=web,api`. **Пресеты** (`1mcp preset create/edit`) → стабильный URL `?preset=name`; правка членства **централизованно**, подключённые клиенты получают новый набор **без правки своих конфигов**. Idle-session pooling.
- **Плюс:** единственный, кто делает **per-project live** красиво — один preset-URL на проект, все агенты наследуют, include/exclude = `preset edit` без трогания файлов.
- **Почему отвергнут:**
  - **Node-демон, висит резидентом** (против C3 — главный стоп).
  - Фильтрация только **server-level** (по тегам), tool-level нет.
  - Гейтвей в пути (осознанный, не скрытый — честнее mcpm, но всё равно прослойка).
  - **Controolloop-челлендж** (см. ниже) показал: для проектов с преобладанием project-local MCP гейтвей централизует крохи при общем failure-domain.

### rulesync (dyoshikawa) — ✅ выбран как слой раздачи
- **Что:** «mcpup, но per-project + мультиагент + больше». Генерит **native** MCP-конфиги per-project из единого `.rulesync/` в 20+ таргетов (Claude Code, Cursor, Codex, OpenCode, **Kilo**, Cline, …). Без враппера. TS/Node, но **CLI на generate, не демон**.
- **Ключевое:** `--input-root <path>` читает source из любого места (dotfiles), `--output-roots` пишет в репу (список — пачкой) → **сидинг без `cp`** (F12). Per-server `targets` (фильтр по агентам — решает Codex-remote). Не затирает неуправляемые записи (repo-local мосты переживают регенерацию, F3). **Бонус:** синкает ещё rules/skills/subagents/ignore (F7).
- **Чего нет:** пресетов/каскада `extends` между папками. Reuse решается `--input-root` (source в dotfiles) + плоскими контекстами.
- **Роль в финале:** per-project native-раздача во все агенты.

### add-mcp (Neon) — ℹ️ заметка
- Императивный one-shot: `npx add-mcp <url|pkg>`, флаги `-a` (агенты), `-g` (глобально), project-level по умолчанию, native-записи. Хорош для «быстро вкинуть один сервер везде», но без source-of-truth. Не финал, держим в уме.

## Ключевые инсайты (что разворачивало решение)

1. **mcpm — скрытый роутер, не менеджер.** Вскрылось только в исходниках: `mcpm run`-враппер + Python-процесс per-клиент. Это и сформировало требование C2 (native/видимое).
2. **Gateway vs syncer — взаимоисключающи.** Нельзя одновременно «чистые конечные конфиги» и «общий процесс/фильтр/live-пресеты». Пришлось выбирать ось.
3. **Per-project реальность.** Ни один тул не убирает per-agent проектные файлы — он лишь меняет их содержимое. 1mcp централизует *членство*; rulesync централизует *источник*. Сам факт N файлов на проект остаётся (по файлу на агента).
4. **Controolloop-челлендж.** Реальный проект показал: большинство его MCP — **project-local HTTP-мосты** (`:8400`, `:3001`), поднимаемые самим проектом. Гонять их через гейтвей = лишний хоп + lifecycle-конфликт + общий failure-domain. Вывод: **гейтвей — только для переиспользуемого stdio-флота; project-local остаётся native.** Это похоронило идею «централизовать всё».
5. **«Системного» MCP-слоя быть не должно.** При радикально разных контекстах агентов общего машинного слоя нет; «система» = инфра (proxy + скиллы), не скоуп.
6. **Каскад — оверинжиниринг при N≈4.** Композер с `extends`/override заменён плоскими контекстами + `--input-root`. Дешевле на порядок.

## Итоговый выбор

**mcp-proxy (Go) + rulesync (Node-CLI) + плоские контексты.**

| Требование | Чем закрыто |
|---|---|
| F1 мультиагент | rulesync (20+ таргетов, вкл. Kilo/Codex/OpenCode) |
| F2 per-project | rulesync per-project `.rulesync/` |
| F3 project-local direct | rulesync не затирает неуправляемые записи |
| F4 reuse флота | proxy (один раз) + `--input-root` |
| F5 tool-фильтр | mcp-proxy `toolFilter` |
| F6 процессы | mcp-proxy (shared child, без утечки) |
| F7 rules/skills | rulesync (бонусом) |
| C1 IaC | proxy `config.json` + контексты в dotfiles |
| C2 native | rulesync пишет native, proxy в пути честно по HTTP |
| C3 без Node-резидента | рантайм = Go-proxy; rulesync — CLI по запросу |
| C4 unix-way | три однозадачные части |
| C5 минимум ручной правки | `mcpctx <ctx>` одной командой |

**Сознательный размен (T1):** членство меняется правкой источника + `mcpctx`, не live-пушем. Это цена за отказ от Node-демона и за видимые native-конфиги — приоритеты, выбранные осознанно по всей ветке.

**Если когда-нибудь важнее станет live-обновление per-project без правки файлов** — это единственный сценарий, где возвращаемся к **1mcp** (ценой Node-демона и server-level фильтра).
